1. Introduction

1.1 Purpose

The NEXUS Database is the second data repository designed to support the operational and financial processes of the Cepat Credit Finance Inc. It serves as a centralized system for storing, managing, and retrieving business data related to loan applications, borrower information, client records, financial transactions, employee records, turnaround time (TAT) monitoring, performance ratings, and other core business functions.

1.2 Scope

This document provides technical and functional documentation for the NEXUS Database. It includes an overview of the database architecture, entity relationships, schema definitions, data dictionary, and table relationships.
The documentation is intended for database administrators, developers, system analysts, data engineers, and other authorized personnel responsible for maintaining, supporting, or integrating with the NEXUS Database.

1.3 Objectives

The primary objectives of the NEXUS Database are:
• To centralize business and financial data within a structured relational database.
• To maintain data accuracy, consistency, and integrity.
• To support day-to-day operational processes and reporting requirements.
• To provide a scalable foundation for future system enhancements and integrations.
• To facilitate efficient data retrieval and analysis for business decision-making.

1.4 Intended Audience

This document is intended for:
• Database Administrators (DBAs)
• Software Developers
• System Analysts
• Data Engineers
• Business Intelligence and Reporting Teams
• Authorized Technical Support Personnel

2. Database Overview

2.1 Functional Areas

Functional Area Description Major Tables
Loan Details/Profiling Maintains detailed information of loan records, including borrower loan information and current loan status. LoanAppHdr, BorrowersInfoHdr, BorrowersEmploymentHdr
Borrower Details Stores detailed borrower profile information and related borrower data. BorrowersInfoHdr, BorrowersEmploymentHdr
Borrower Financial Details Contains borrower financial attributes such as net disposable income, income sources, and other financial capacity indicators used for profiling. BorrowersNdi, OwnedAssets
Employee Management Stores employee information and system user records. UsersAccounts
Employee Performance Monitoring Tracks turnaround time (TAT), ratings, and operational performance metrics. Ratings, Remarks, UsersAccounts
Loan Status History & Tracking Tracks the history of loan status changes throughout the loan lifecycle, including the current and previous process stages. It also records internal and external remarks as well as employees involved in each stage of the loan process for audit and traceability purposes. LoanAppHdr, Remarks

2.3 Naming Conventions

Convention Description Convention Description
Hdr Header table containing primary or master records. Dtl Detail table containing child or transactional records.
PK Primary Key used to uniquely identify a record. FK Foreign Key used to establish relationships between tables.
ModUser User ID of the one who modified the record. ModDate Date and time when the record was last modified.
Status Indicates the current process status of the Loan. PB Principal Borrower for the Loan.
CB Co-Borrower for the Loan. LC Loan Consultant for the Loan.
CRA Credit Review Associate. CRO Credit Review Officer.
CreMan Credit Manager. NDI Net Disposable Income.

5. Data Rules / Transformation Logic

1: Table Aliasing Convention

Tables follow a standardized aliasing convention where the alias is formed by taking the first letter of each word in the table name.

  • LoanAppHdr → LAH
  • BorrowerInfoHdr → BIH

2: Source of Data for Financial Values

Financial-related data values such as monetary amounts and similar financial attributes should be sourced from the SOFIA database as the primary source. Data should only be retrieved from the NEXUS database if explicitly specified or required by the report or query definition.

3: “As Of” Date Filtering Logic

When a request specifies data “As of (date)”, the data extraction must cover the full available range starting from the system baseline date up to the specified “As of” date.

4: Standard Naming for Borrower Roles

For reporting outputs, borrower role labels must use OFW instead of PB (Principal Borrower) and Beneficiary instead of CB (Co-Borrower) to avoid confusion in generated reports.

5: Free Text Trimming

All free text inputs, especially names, must be trimmed to remove leading and trailing spaces before storage or use in reporting to ensure data consistency and avoid duplication caused by whitespace variations. When extracting or displaying borrower names, the format must follow "Last Name, First Name Middle Name".
